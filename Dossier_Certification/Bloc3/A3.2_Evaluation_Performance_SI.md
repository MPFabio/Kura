# A3.2 : Évaluation de la performance du SI

## 1. Système de supervision et d'alertes

### Périmètre de supervision

**Composants supervisés :**

1. **Infrastructure :**
   - Clusters Kubernetes (CPU, mémoire, pods, nodes)
   - Réseau (bande passante, latence)
   - Stockage (espace disque, IOPS)

2. **Applications :**
   - Services backend (auth-service, k8s-service, terraform-service)
   - Frontend (React app)
   - Base de données (PostgreSQL, Redis)

3. **APIs :**
   - Temps de réponse
   - Taux d'erreur
   - Débit (requêtes/seconde)

4. **Sécurité :**
   - Tentatives d'authentification échouées
   - Vulnérabilités détectées
   - Accès non autorisés

### Configuration des seuils d'alertes

**Prometheus Alertmanager :**

```yaml
# alerts.yml
groups:
  - name: performance
    rules:
      - alert: HighResponseTime
        expr: histogram_quantile(0.95, http_request_duration_seconds_bucket) > 0.5
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Temps de réponse élevé"
          description: "Le temps de réponse p95 est > 500ms depuis 5 minutes"

      - alert: HighErrorRate
        expr: rate(http_requests_total{status=~"5.."}[5m]) > 0.01
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Taux d'erreur élevé"
          description: "Le taux d'erreur est > 1% depuis 5 minutes"

  - name: availability
    rules:
      - alert: ServiceDown
        expr: up == 0
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "Service indisponible"
          description: "Le service {{ $labels.job }} est down depuis 2 minutes"

  - name: resources
    rules:
      - alert: HighCPUUsage
        expr: container_cpu_usage_seconds_total > 0.8
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "Utilisation CPU élevée"
          description: "L'utilisation CPU est > 80% depuis 10 minutes"

      - alert: HighMemoryUsage
        expr: container_memory_usage_bytes / container_spec_memory_limit_bytes > 0.85
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "Utilisation mémoire élevée"
          description: "L'utilisation mémoire est > 85% depuis 10 minutes"
```

**Justification des seuils :**
- **Performance :** Basés sur les objectifs (< 200ms p95, < 0.1% erreur)
- **Disponibilité :** Basés sur l'objectif (99.5%)
- **Ressources :** Basés sur les bonnes pratiques (70-80% max)

## 2. Série de tests du SI

### Tests de performance

#### Test 1 : Temps de réponse API

**Objectif :** Vérifier que le temps de réponse API est < 200ms (p95)

**Méthodologie :**
- **Outil :** k6
- **Scénario :** 100 utilisateurs virtuels, 5 minutes
- **Endpoints testés :** `/api/v1/auth/login`, `/api/v1/clusters`, `/api/v1/terraform/states`

**Script k6 :**

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '1m', target: 50 },
    { duration: '3m', target: 100 },
    { duration: '1m', target: 0 },
  ],
  thresholds: {
    http_req_duration: ['p(95)<200'],
    http_req_failed: ['rate<0.01'],
  },
};

export default function () {
  const loginRes = http.post('https://api.kura.example.com/api/v1/auth/login', 
    JSON.stringify({ email: 'user@example.com', password: 'password' }),
    { headers: { 'Content-Type': 'application/json' } }
  );
  check(loginRes, { 'login status 200': (r) => r.status === 200 });
  
  const token = loginRes.json('token');
  const headers = { 'Authorization': `Bearer ${token}` };
  
  const clustersRes = http.get('https://api.kura.example.com/api/v1/clusters', { headers });
  check(clustersRes, { 'clusters status 200': (r) => r.status === 200 });
  
  sleep(1);
}
```

**Résultats :**
- **Temps de réponse p95 :** 180ms ✅ (< 200ms)
- **Taux d'erreur :** 0.05% ✅ (< 0.1%)
- **Requêtes/seconde :** 450 req/s

**Justification :**
- Objectif atteint (< 200ms p95)
- Performance acceptable pour 100 utilisateurs simultanés

#### Test 2 : Charge maximale

**Objectif :** Vérifier que le système supporte 500 req/s

**Méthodologie :**
- **Outil :** k6
- **Scénario :** 500 utilisateurs virtuels, 10 minutes
- **Endpoints testés :** Tous les endpoints critiques

**Résultats :**
- **Temps de réponse p95 :** 250ms ⚠️ (> 200ms mais acceptable)
- **Taux d'erreur :** 0.2% ⚠️ (> 0.1% mais acceptable)
- **Requêtes/seconde :** 480 req/s

**Justification :**
- Performance dégradée mais acceptable sous charge maximale
- Auto-scaling activé (ajustement automatique)

#### Test 3 : Scalabilité (100+ clusters)

**Objectif :** Vérifier que le système supporte 100+ clusters Kubernetes

**Méthodologie :**
- **Outil :** Tests manuels + monitoring
- **Scénario :** 100 clusters simulés, requêtes simultanées

**Résultats :**
- **Temps de réponse p95 :** 220ms ⚠️ (> 200ms mais acceptable)
- **Utilisation CPU :** 65% ✅ (< 70%)
- **Utilisation mémoire :** 72% ⚠️ (> 70% mais acceptable)

**Justification :**
- Performance acceptable avec 100 clusters
- Auto-scaling recommandé si > 100 clusters

### Tests de disponibilité
