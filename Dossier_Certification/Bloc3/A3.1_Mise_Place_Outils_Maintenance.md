# A3.1 : Mise en place des outils de maintenance du SI

## 1. Sélection des outils de maintenance

### Étude comparative des outils de maintenance

#### Outils de gestion de code source

**GitHub :**
- **Avantages :** Standard de l'industrie, intégration CI/CD, communauté
- **Inconvénients :** Coûts pour les organisations privées
- **Adaptation :** ✅ Oui (déjà utilisé)

**GitLab :**
- **Avantages :** Open-source, CI/CD intégré, gestion de projet
- **Inconvénients :** Moins de fonctionnalités que GitHub
- **Adaptation :** ⚠️ Partielle (alternative)

**Bitbucket :**
- **Avantages :** Intégration Atlassian, gratuit pour petites équipes
- **Inconvénients :** Moins populaire, fonctionnalités limitées
- **Adaptation :** ❌ Non

**Choix :** **GitHub** (déjà utilisé, standard)

#### Outils de bug tracking

**Jira :**
- **Avantages :** Puissant, personnalisable, intégration avec autres outils
- **Inconvénients :** Complexe, coûteux
- **Adaptation :** ✅ Oui (gestion de projet)

**GitHub Issues :**
- **Avantages :** Intégré à GitHub, simple, gratuit
- **Inconvénients :** Fonctionnalités limitées vs Jira
- **Adaptation :** ⚠️ Partielle (petits projets)

**Linear :**
- **Avantages :** Moderne, rapide, UX excellente
- **Inconvénients :** Moins de fonctionnalités, coûteux
- **Adaptation :** ⚠️ Partielle

**Choix :** **Jira** (puissant, déjà utilisé)

#### Outils de CI/CD

**GitHub Actions :**
- **Avantages :** Intégré à GitHub, gratuit pour publics, nombreux templates
- **Inconvénients :** Limites pour les projets privés
- **Adaptation :** ✅ Oui (déjà utilisé)

**GitLab CI :**
- **Avantages :** Intégré à GitLab, puissant
- **Inconvénients :** Nécessite GitLab
- **Adaptation :** ⚠️ Partielle

**Jenkins :**
- **Avantages :** Open-source, très personnalisable, nombreux plugins
- **Inconvénients :** Complexe, maintenance lourde
- **Adaptation :** ⚠️ Partielle (si besoin de plus de contrôle)

**Choix :** **GitHub Actions** (intégré, simple)

#### Outils de tests

**Go testing :**
- **Avantages :** Intégré à Go, simple
- **Inconvénients :** Fonctionnalités limitées
- **Adaptation :** ✅ Oui (tests unitaires)

**Testify :**
- **Avantages :** Bibliothèque Go, assertions, mocks
- **Inconvénients :** Dépendance externe
- **Adaptation :** ✅ Oui (tests unitaires, intégration)

**Cypress :**
- **Avantages :** Tests e2e, moderne, bon debugging
- **Inconvénients :** Coûteux pour les équipes
- **Adaptation :** ✅ Oui (tests e2e frontend)

**Choix :** **Go testing + Testify** (backend), **Cypress** (frontend)

#### Outils de monitoring

**Prometheus :**
- **Avantages :** Open-source, standard, puissant
- **Inconvénients :** Pas de stockage long terme
- **Adaptation :** ✅ Oui (déjà utilisé)

**Grafana :**
- **Avantages :** Visualisation, dashboards, open-source
- **Inconvénients :** Nécessite une source de données
- **Adaptation :** ✅ Oui (déjà utilisé)

**Choix :** **Prometheus + Grafana** (déjà utilisés)

### Synthèse de l'étude comparative

| Catégorie | Outil choisi | Justification |
|-----------|--------------|---------------|
| Gestion de code source | GitHub | Standard, intégration CI/CD |
| Bug tracking | Jira | Puissant, gestion de projet |
| CI/CD | GitHub Actions | Intégré, simple |
| Tests backend | Go testing + Testify | Standard Go, assertions |
| Tests frontend | Cypress | Tests e2e, moderne |
| Monitoring | Prometheus + Grafana | Standard, open-source |

**Bénéfices :**
- **Cohérence :** Outils standards, bien documentés
- **Intégration :** GitHub Actions avec GitHub, Jira avec autres outils
- **Coûts :** Majoritairement open-source ou déjà utilisés
- **Maintenance :** Outils matures, communauté active

### Adaptation au contexte de l'organisation

**Contexte :**
- Équipe de 4-5 personnes
- Projet cloud-native (Kubernetes)
- Technologies : Go, React, Kubernetes
- Budget limité (privilégier open-source)

**Adaptation :**
- **GitHub :** Déjà utilisé, standard
- **Jira :** Gestion de projet, suivi des tâches
- **GitHub Actions :** Intégré, gratuit pour publics
- **Prometheus + Grafana :** Standard, open-source

**Bénéfices démontrés :**
- Réduction des coûts (open-source)
- Simplification (moins d'outils)
- Intégration (écosystème cohérent)
- Maintenabilité (outils standards)

## 2. Déploiement des outils de maintenance

### Configuration et déploiement

#### Outil 1 : GitHub (Gestion de code source)

**Configuration :**

**1. Organisation et repositories :**
- Organisation : `Kura`
- Repositories :
  - `kura/kura-backend` (services Go)
  - `kura/kura-frontend` (React)
  - `kura/kura-infrastructure` (Kubernetes, Docker)

**2. Branches :**
- `main` : Production
- `develop` : Développement
- `feature/*` : Nouvelles fonctionnalités
- `hotfix/*` : Corrections urgentes

**3. Protection des branches :**
- `main` : Pull request requis, review obligatoire, tests passés
- `develop` : Pull request requis, tests passés

**4. Webhooks :**
- Intégration avec Jira (création de tickets)
- Intégration avec GitHub Actions (CI/CD)

**Capture d'écran : Configuration des branches**

```
Repository: kura/kura-backend
├── Settings
│   ├── Branches
│   │   ├── Branch protection rules
│   │   │   ├── main
│   │   │   │   ├── Require pull request reviews: ✅
│   │   │   │   ├── Required reviewers: 1
│   │   │   │   ├── Require status checks: ✅
│   │   │   │   │   ├── Test: ✅
│   │   │   │   │   └── Lint: ✅
│   │   │   │   └── Require branches to be up to date: ✅
```

**Commentaire :**
- Protection de la branche `main` pour garantir la qualité
- Review obligatoire pour éviter les erreurs
- Tests obligatoires pour garantir la stabilité

**Utilisation :**
- Développement sur branches `feature/*`
- Pull request vers `develop` puis `main`
- Tests automatiques (GitHub Actions)
- Review par un pair

#### Outil 3 : GitHub Actions (CI/CD)

**Configuration :**

**1. Workflows :**
- `test.yml` : Tests unitaires, intégration
- `build.yml` : Build et push d'images Docker
- `deploy.yml` : Déploiement sur Kubernetes

**2. Triggers :**
- Push sur `develop`, `main`
- Pull request
- Manuelle (workflow_dispatch)

**3. Jobs :**
- `test` : Tests Go, React
- `build` : Build Docker images
- `deploy` : Déploiement Kubernetes (staging, prod)

**Capture d'écran : Configuration du workflow CI/CD**

```yaml
# .github/workflows/test.yml
name: Tests

on:
  push:
    branches: [develop, main]
  pull_request:
    branches: [develop, main]

jobs:
  test-backend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-go@v4
        with:
          go-version: '1.22'
      - run: go test ./... -v -coverprofile=coverage.out
      - run: go tool cover -html=coverage.out -o coverage.html
      - uses: codecov/codecov-action@v3
        with:
          file: ./coverage.out
```

**Commentaire :**
- Tests automatiques à chaque push/PR
- Couverture de code (objectif > 80%)
- Intégration avec Codecov pour le suivi

**Utilisation :**
- Exécution automatique (push, PR)
- Rapport de couverture (Codecov)
- Blocage des PR si tests échouent

#### Outil 4 : Prometheus + Grafana (Monitoring)

**Configuration :**

**1. Prometheus :**
- Scrape interval : 15s
- Retention : 30 jours
- Alertmanager : Configuré

**2. Grafana :**
- Dashboards : Kubernetes, Applications, Infrastructure
- Alertes : Email, Slack
- Data sources : Prometheus, PostgreSQL

**Capture d'écran : Configuration Prometheus**

```yaml
# prometheus.yml
global:
  scrape_interval: 15s
  retention: 30d

scrape_configs:
  - job_name: 'kubernetes'
    kubernetes_sd_configs:
      - role: pod
    relabel_configs:
      - source_labels: [__meta_kubernetes_pod_name]
        target_label: pod
```

**Commentaire :**
- Scrape automatique des pods Kubernetes
- Métriques collectées toutes les 15s
- Rétention de 30 jours pour l'historique

**Utilisation :**
- Monitoring en temps réel
- Alertes (seuils définis)
- Dashboards (visualisation)

### Priorisation des outils

**Priorité 1 (Critique) :**
- GitHub (gestion de code source)
- GitHub Actions (CI/CD)
- Prometheus + Grafana (monitoring)

**Priorité 2 (Élevée) :**
- Jira (bug tracking)
- Tests automatisés (Go testing, Cypress)

**Priorité 3 (Moyenne) :**
- Outils de sécurité (Snyk, Dependabot)
- Outils de documentation (Confluence)

### Processus de maintenance

**Maintenance préventive :**
- Mise à jour régulière des dépendances (Dependabot)
- Scan de vulnérabilités (Snyk)
- Tests de régression (CI/CD)

**Maintenance corrective :**
- Correction des bugs (Jira)
- Hotfixes (branches `hotfix/*`)
- Déploiement rapide (GitHub Actions)

**Maintenance évolutive :**
- Nouvelles fonctionnalités (branches `feature/*`)
- Améliorations (sprints)
- Optimisations (performance, coûts)
