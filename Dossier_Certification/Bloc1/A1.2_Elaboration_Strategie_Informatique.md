# A1.2 : Élaboration de la stratégie informatique

## 1. Diagnostic des besoins

### Synthèse de l'analyse préalable

À partir du diagnostic réalisé dans A1.1, les besoins principaux sont :

1. **Centralisation des outils DevOps**
   - Interface unique pour Kubernetes, Terraform, Ansible, CI/CD
   - Réduction de la charge cognitive

2. **Authentification unifiée**
   - JWT centralisé
   - Gestion des rôles homogène

3. **Corrélation d'événements**
   - Bus d'événements (Kafka)
   - Traçabilité de bout en bout

4. **Observabilité intégrée**
   - Métriques, logs, alertes centralisées
   - Dashboards unifiés

### Analyse des besoins techniques

**Besoins fonctionnels :**
- Gestion des clusters Kubernetes (CRUD, actions, terminal)
- Gestion des états Terraform (upload, visualisation, synchronisation, drift)
- Gestion des jobs Ansible (exécution, monitoring)
- Gestion des pipelines CI/CD (visualisation, déclenchement)
- Authentification et autorisation
- Observabilité (métriques, logs, alertes)

**Besoins non-fonctionnels :**
- Performance : Temps de réponse < 200ms (p95)
- Disponibilité : 99.5%
- Scalabilité : Support de 100+ clusters
- Sécurité : Conformité RGPD, ISO 27001
- Maintenabilité : Documentation, tests automatisés

## 2. Stratégie du SI cible

### Vision stratégique

**Objectif :** Créer une plateforme DevOps unifiée qui :
- Centralise la gestion de tous les outils DevOps
- Offre une expérience utilisateur cohérente
- Facilite les diagnostics cross-systèmes
- Améliore la productivité des équipes

### Architecture cible

```
┌─────────────────────────────────────────────────────────┐
│                    Frontend React                        │
│              (Interface utilisateur unique)              │
└───────────────────────┬─────────────────────────────────┘
                        │
┌───────────────────────▼─────────────────────────────────┐
│              API Gateway (Kong)                         │
│    (Routage, Authentification, Rate Limiting)           │
└───────┬───────────┬───────────┬───────────┬───────────┘
        │           │           │           │
┌───────▼───┐ ┌─────▼─────┐ ┌───▼─────┐ ┌───▼─────┐
│ Auth      │ │ K8s       │ │ Terraform│ │ Ansible │
│ Service   │ │ Service   │ │ Service  │ │ Service │
└───────┬───┘ └─────┬─────┘ └───┬─────┘ └───┬─────┘
        │           │           │           │
        └───────────┴───────────┴───────────┘
                        │
        ┌───────────────▼───────────────┐
        │      Kafka Event Bus          │
        │  (Corrélation d'événements)   │
        └───────────────┬───────────────┘
                        │
        ┌───────────────▼───────────────┐
        │  Alert Service │ Metrics      │
        │  Service       │ Service       │
        └───────────────────────────────┘
                        │
┌───────────────────────┴───────────────────────────────┐
│  PostgreSQL  │  Redis  │  Prometheus  │  Grafana     │
│  (Données)   │ (Cache) │ (Métriques)  │ (Viz)        │
└────────────────────────────────────────────────────────┘
```

### Principes d'architecture

1. **Microservices :** Séparation des responsabilités par domaine
   - `auth-service` : Authentification
   - `k8s-service` : Gestion Kubernetes
   - `terraform-service` : Gestion Terraform
   - `ansible-service` : Gestion Ansible
   - `pipeline-service` : Gestion CI/CD
   - `alert-service` : Alertes
   - `metrics-service` : Métriques

2. **API Gateway :** Point d'entrée unique (Kong)
   - Routage des requêtes
   - Authentification centralisée
   - Rate limiting
   - Logging et monitoring

3. **Event-driven :** Communication asynchrone (Kafka)
   - Découplage des services
   - Corrélation d'événements
   - Scalabilité horizontale

4. **Cloud-native :** Containers, Kubernetes, multi-cloud
   - Déploiement sur Kubernetes
   - Support multi-cloud (AWS, Azure, GCP)
   - Auto-scaling

5. **Observabilité :** Métriques, logs, traces
   - Prometheus (métriques)
   - Grafana (visualisation)
   - Logs centralisés

### Solutions techniques proposées

#### Backend
- **Langage :** Go
  - Performance élevée
  - Concurrence native (goroutines)
  - Écosystème Kubernetes mature
- **Framework :** Gin
  - Léger et performant
  - Middleware intégré
- **Base de données :** PostgreSQL 15
  - ACID, transactions
  - Support JSON
- **Cache :** Redis 7
  - Performance
  - Persistance (AOF + RDB)

#### Frontend
- **Framework :** React + TypeScript
  - Écosystème riche
  - Type safety
- **State management :** Redux Toolkit
- **UI :** Material-UI ou Ant Design

#### Infrastructure
- **Orchestration :** Kubernetes 1.28+
- **API Gateway :** Kong 3.4
- **Event Bus :** Kafka + Zookeeper
- **Monitoring :** Prometheus + Grafana
- **Containerisation :** Docker

#### Intégrations
- **Kubernetes :** client-go
- **Terraform :** Terraform Cloud API, Terraform SDK
- **Ansible :** Ansible Tower API
- **CI/CD :** GitHub Actions API, GitLab CI API, Jenkins API

### Justification des choix techniques

**Go pour le backend :**
- Performance élevée (compilé, pas d'interpréteur)
- Concurrence native (goroutines)
- Écosystème Kubernetes mature (client-go)
- Simplicité de déploiement (binaire unique)

**React + TypeScript pour le frontend :**
- Écosystème riche (bibliothèques, outils)
- Type safety (réduction des erreurs)
- Performance (virtual DOM)
- Maintenabilité (composants réutilisables)

**Kong comme API Gateway :**
- Open-source
- Performant et scalable
- Plugins (authentification, rate limiting, logging)
- Support multi-protocole (HTTP, gRPC, WebSocket)

**Kafka comme event bus :**
- Haute performance (débit élevé)
- Durabilité (persistance des messages)
- Scalabilité horizontale
- Découplage des services

**PostgreSQL comme base de données :**
- ACID (fiabilité)
- Support JSON (flexibilité)
- Performances (index, requêtes optimisées)
- Écosystème mature

**Kubernetes pour l'orchestration :**
- Standard de l'industrie
- Auto-scaling, auto-healing
- Multi-cloud
- Écosystème riche

### Adéquation avec la stratégie du SI

Les solutions techniques proposées sont en adéquation avec la stratégie du SI car :

1. **Scalabilité :** Architecture microservices + Kubernetes permet la mise à l'échelle horizontale
2. **Disponibilité :** Haute disponibilité via Kubernetes (multi-pods, health checks)
3. **Sécurité :** Authentification centralisée (Kong), chiffrement (TLS), audit
4. **Maintenabilité :** Séparation des responsabilités, tests automatisés, documentation
5. **Performance :** Cache (Redis), optimisations (Go), monitoring (Prometheus)

## 3. Politique de sous-traitance

### Analyse de la politique de sous-traitance

**Stratégie :** Développement interne avec recours ponctuel à la sous-traitance

#### Développement interne

**Équipe interne :**
- 2 développeurs Go (backend)
- 1 développeur React/TypeScript (frontend)
- 1 DevOps/Architecte
- 1 Product Owner (temps partiel)

**Avantages :**
- Maîtrise du code source
- Connaissance métier
- Réactivité
- Coûts maîtrisés

#### Sous-traitance ponctuelle

**Cas d'usage :**
1. **Audit de sécurité :** Expertise externe (conformité ISO 27001)
2. **Tests de charge :** Validation de la scalabilité
3. **Formation :** Montée en compétences (Kubernetes avancé, Kafka)
4. **Design UI/UX :** Interface utilisateur (si besoin)

**Critères de sélection :**
- Expertise technique
- Références
- Coûts
- Disponibilité
- Conformité (RGPD, sécurité)

**Gestion de la sous-traitance :**
- Contrats clairs (SLA, livrables, délais)
- Suivi régulier (points hebdomadaires)
- Validation des livrables
- Documentation partagée

### Organisation optimale

**Recommandation :** Développement interne avec sous-traitance ponctuelle pour :
- Audit de sécurité (annuel)
- Tests de charge (avant production)
- Formation (ponctuelle)

**Justification :**
- Maîtrise du code source et de l'architecture
- Réduction des risques (sécurité, dépendance)
- Coûts optimisés
- Flexibilité

## 4. Plan de Reprise d'Activité (PRA) et Plan de Continuité d'Activité (PCA)

### Processus critiques

**P1 : Authentification**
- **Criticité :** Critique
- **Impact :** Blocage de tous les utilisateurs
- **RTO :** 1 heure
- **RPO :** 0

**P2 : Gestion des clusters Kubernetes**
- **Criticité :** Critique
- **Impact :** Impossibilité de déployer/monitorer
- **RTO :** 2 heures
- **RPO :** 15 minutes

**P3 : Gestion des états Terraform**
- **Criticité :** Élevée
- **Impact :** Perte de synchronisation infrastructure
- **RTO :** 4 heures
- **RPO :** 1 heure

**P4 : Observabilité**
- **Criticité :** Moyenne
- **Impact :** Perte de visibilité
- **RTO :** 8 heures
- **RPO :** 1 heure

### Scénarios critiques

**S1 : Panne de l'API Gateway (Kong)**
- **Description :** Indisponibilité de tous les services
- **Probabilité :** Faible
- **Impact :** Critique
- **RTO :** 1 heure
- **RPO :** 0

**S2 : Panne de la base de données PostgreSQL**
- **Description :** Perte d'accès aux données
- **Probabilité :** Faible
- **Impact :** Critique
- **RTO :** 4 heures
- **RPO :** 1 heure

**S3 : Panne de Kafka**
- **Description :** Perte de corrélation d'événements
- **Probabilité :** Faible
- **Impact :** Élevé
- **RTO :** 2 heures
- **RPO :** 15 minutes

**S4 : Panne d'un cloud provider**
- **Description :** Perte d'accès aux ressources cloud
- **Probabilité :** Très faible
- **Impact :** Critique
- **RTO :** 24 heures
- **RPO :** 1 heure

### Plan de Reprise d'Activité (PRA)

#### PRA - Panne API Gateway (Kong)

**Objectifs :**
- RTO : 1 heure
- RPO : 0

**Procédure de reprise :**

1. **Détection (0-5 min)**
   - Monitoring Prometheus détecte la panne
   - Alerte envoyée (email, Slack)
   - Équipe on-call notifiée

2. **Diagnostic (5-15 min)**
   - Vérification de l'état des pods Kong (kubectl)
   - Consultation des logs (kubectl logs)
   - Identification de la cause (pod crash, node failure, etc.)

3. **Action corrective (15-45 min)**
   - **Si pod crash :** Redémarrage automatique (Kubernetes)
   - **Si node failure :** Bascule vers node de secours
   - **Si configuration :** Restauration depuis Git
   - **Si instance de secours :** Bascule manuelle

4. **Vérification (45-60 min)**
   - Tests de santé des services (health checks)
   - Vérification du routage (test API)
   - Validation de l'authentification
   - Communication aux utilisateurs

**Ressources nécessaires :**
- Instance Kong de secours (standby, multi-zone)
- Configuration sauvegardée (Git)
- Monitoring (Prometheus, alertes)
- Documentation des procédures

**Acteurs :**
- DevOps on-call (détection, diagnostic)
- Équipe technique (action corrective)
- Product Owner (communication)

#### PRA - Panne base de données PostgreSQL

**Objectifs :**
- RTO : 4 heures
- RPO : 1 heure

**Procédure de reprise :**

1. **Détection (0-5 min)**
   - Monitoring Prometheus détecte la panne
   - Alerte envoyée
   - Équipe on-call notifiée

2. **Diagnostic (5-30 min)**
   - Vérification de l'état du pod PostgreSQL
   - Consultation des logs
   - Identification de la cause (corruption, node failure, etc.)

3. **Action corrective (30 min - 3h)**
   - **Si pod crash :** Redémarrage (Kubernetes)
   - **Si corruption :** Restauration depuis backup
   - **Si node failure :** Bascule vers instance de secours
   - **Si perte de données :** Restauration depuis backup (dernière sauvegarde)

4. **Vérification (3h - 4h)**
   - Tests de connexion (psql)
   - Vérification de l'intégrité des données
   - Tests des services dépendants
   - Communication aux utilisateurs

**Ressources nécessaires :**
- Backups PostgreSQL (quotidien, rétention 30 jours)
- Instance de secours (standby, réplication)
- Procédures de restauration documentées
- Monitoring (Prometheus, alertes)

**Acteurs :**
- DevOps on-call (détection, diagnostic)
- DBA / Équipe technique (restauration)
- Product Owner (communication)

### Plan de Continuité d'Activité (PCA)

#### PCA - Haute disponibilité

**Objectifs :**
- Disponibilité : 99.5%
- Redondance : Multi-zone, multi-cloud

**Mesures préventives :**

1. **Infrastructure :**
   - **Déploiement multi-zone :** Services déployés sur plusieurs zones de disponibilité
   - **Load balancing :** Kong et Kubernetes répartissent la charge
   - **Auto-scaling :** Kubernetes HPA (Horizontal Pod Autoscaler) ajuste le nombre de pods
   - **Health checks :** Vérification de santé des services (liveness, readiness)

2. **Données :**
   - **Réplication PostgreSQL :** Master-slave (streaming replication)
   - **Réplication Redis :** Sentinel pour haute disponibilité
   - **Backups automatisés :** Quotidien, rétention 30 jours
   - **Backups multi-région :** Copies dans plusieurs régions

3. **Monitoring :**
   - **Alertes Prometheus :** Seuils définis (disponibilité, performance, erreurs)
   - **Dashboard Grafana :** Vue d'ensemble en temps réel
   - **Notifications :** Email, Slack, PagerDuty
   - **On-call :** Rotation des équipes

4. **Sécurité :**
   - **Chiffrement :** TLS (en transit), chiffrement au repos
   - **Authentification :** JWT, refresh tokens
   - **Audit :** Logs de toutes les actions
   - **Conformité :** RGPD, ISO 27001

#### Indicateurs de supervision

**RTO (Recovery Time Objective) :**
- API Gateway : 1 heure
- Base de données : 4 heures
- Kafka : 2 heures
- Cloud provider : 24 heures

**RPO (Recovery Point Objective) :**
- API Gateway : 0 (pas de données)
- Base de données : 1 heure (backup quotidien)
- Kafka : 15 minutes (buffer)
- Cloud provider : 1 heure

**Métriques de disponibilité :**
- **Uptime :** 99.5% (objectif)
- **Temps de réponse API :** < 200ms (p95)
- **Taux d'erreur :** < 0.1%

**Seuils d'alerte :**
- **Disponibilité < 99% :** Alerte critique
- **Temps de réponse > 500ms :** Alerte warning
- **Taux d'erreur > 1% :** Alerte critique
- **Utilisation CPU > 80% :** Alerte warning
- **Utilisation mémoire > 85% :** Alerte warning
- **Espace disque < 20% :** Alerte warning

**Tableau de bord de supervision :**

| Métrique | Seuil Warning | Seuil Critique | Action |
|----------|---------------|----------------|--------|
| Disponibilité | < 99.5% | < 99% | Investigation immédiate |
| Temps de réponse API | > 300ms | > 500ms | Optimisation |
| Taux d'erreur | > 0.5% | > 1% | Investigation immédiate |
| Utilisation CPU | > 70% | > 80% | Scale up |
| Utilisation mémoire | > 75% | > 85% | Scale up |
| Espace disque | < 30% | < 20% | Nettoyage, extension |

## 5. Identification des risques du SI

### Risques identifiés

#### Risques techniques

**R1 : Complexité de l'intégration**
- **Probabilité :** Moyenne
- **Impact :** Élevé
- **Mitigation :** Développement incrémental, tests d'intégration, abstraction des APIs

**R2 : Performance à grande échelle**
- **Probabilité :** Moyenne
- **Impact :** Moyen
- **Mitigation :** Tests de charge, optimisation, cache (Redis), mise à l'échelle horizontale

**R3 : Évolution des APIs externes**
- **Probabilité :** Faible
- **Impact :** Élevé
- **Mitigation :** Abstraction, versioning, monitoring des changements d'APIs

#### Risques de sécurité

**R4 : Vulnérabilités dans les dépendances**
- **Probabilité :** Moyenne
- **Impact :** Élevé
- **Mitigation :** Scan de vulnérabilités (Snyk, Dependabot), mise à jour régulière

**R5 : Exposition d'APIs**
- **Probabilité :** Faible
- **Impact :** Élevé
- **Mitigation :** API Gateway (Kong), authentification, rate limiting, WAF

**R6 : Fuite de données sensibles**
- **Probabilité :** Faible
- **Impact :** Critique
- **Mitigation :** Chiffrement (TLS, au repos), gestion des secrets (Vault), audit

#### Risques opérationnels

**R7 : Panne d'infrastructure**
- **Probabilité :** Faible
- **Impact :** Critique
- **Mitigation :** Haute disponibilité (multi-zone), backups, PRA/PCA

**R8 : Perte de données**
- **Probabilité :** Très faible
- **Impact :** Critique
- **Mitigation :** Backups automatisés, réplication, tests de restauration

### Analyse des risques

Les risques sont pris en compte dans :
- **Architecture :** Haute disponibilité, redondance
- **Sécurité :** Chiffrement, authentification, audit
- **Monitoring :** Alertes, dashboards
- **Procédures :** PRA/PCA, documentation
