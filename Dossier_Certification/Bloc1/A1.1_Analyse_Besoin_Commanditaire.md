## 3. Analyse de la demande, des objectifs et des enjeux

### Problématique identifiée

Les équipes DevOps utilisent actuellement une multitude d'outils spécialisés mais isolés :
- **Portails Kubernetes** (Lens, Octant)
- **Outils Terraform** (Terraform Cloud, Atlantis)
- **Interfaces Ansible** (Ansible Tower/AWX)
- **Outils CI/CD** (GitHub Actions, GitLab CI, Jenkins)
- **Stacks d'observabilité** (Prometheus/Grafana)

**Conséquences :**
- Charge cognitive élevée (navigation entre plusieurs interfaces)
- Difficulté à avoir une vision transversale
- Authentification et rôles dupliqués dans chaque outil
- Manque de corrélation entre événements

### Enjeux du projet

**Enjeux techniques :**
- Intégration de multiples APIs (Kubernetes, Terraform Cloud, Ansible Tower, etc.)
- Gestion de la complexité d'une architecture microservices
- Performance et scalabilité de la plateforme
- Sécurité et conformité (RGPD, ISO 27001)

**Enjeux organisationnels :**
- Adoption par les équipes DevOps
- Formation des utilisateurs
- Changement de pratiques
- Support et maintenance

**Enjeux financiers :**
- Coût de développement
- Coût d'infrastructure (cloud, licences)
- ROI attendu (gain de productivité)

**Enjeux stratégiques :**
- Différenciation concurrentielle
- Modernisation de l'infrastructure
- Préparation à la croissance

### Objectifs structurés

L'analyse de la demande permet de définir les objectifs suivants :

1. **Objectif fonctionnel principal :**
   Créer une plateforme DevOps unifiée offrant une vue centralisée sur Kubernetes, Terraform, Ansible et les pipelines CI/CD.

2. **Objectifs techniques :**
   - Architecture microservices avec API Gateway (Kong)
   - Bus d'événements (Kafka) pour la corrélation
   - Authentification centralisée (JWT)
   - Observabilité intégrée (Prometheus, Grafana)

3. **Objectifs de performance :**
   - Temps de réponse API < 200ms (p95)
   - Disponibilité 99.5%
   - Support de 100+ clusters Kubernetes simultanés

4. **Objectifs de qualité :**
   - Documentation complète
   - Tests automatisés (couverture > 80%)
   - Sécurité renforcée (OWASP Top 10)

- Architecture microservices éprouvée

## 7. Analyse des risques informatiques

### Méthodologie d'analyse des risques

Pour réaliser une analyse complète et structurée des risques, plusieurs méthodes complémentaires sont utilisées :

#### Méthode harmonisée d'analyse des risques (MEHARI)

**MEHARI** est une méthode d'analyse des risques développée par le CLUSIF (Club de la Sécurité de l'Information Français). Elle permet d'évaluer les risques de manière structurée et harmonisée.

**Application au projet :**

**Étape 1 : Identification des actifs**
- Actifs informationnels : Données utilisateurs, états Terraform, configurations
- Actifs applicatifs : Services backend, frontend, APIs
- Actifs infrastructure : Clusters Kubernetes, bases de données, réseaux

**Étape 2 : Identification des menaces**
- Menaces techniques : Vulnérabilités, pannes, erreurs de configuration
- Menaces humaines : Erreurs, malveillance, négligence
- Menaces organisationnelles : Changements, manque de compétences

**Étape 3 : Évaluation de la vulnérabilité**
- Vulnérabilités identifiées : Dépendances non mises à jour, configuration par défaut, absence de monitoring
- Niveau de vulnérabilité : Élevé, Moyen, Faible

**Étape 4 : Calcul du niveau de risque**
- Niveau de risque = Menace × Vulnérabilité × Impact
- Classification : Critique, Élevé, Moyen, Faible

**Résultat :** Identification structurée de 11 risques principaux, classés par niveau de criticité.

#### EBIOS Risk Manager

**EBIOS Risk Manager** (Expression des Besoins et Identification des Objectifs de Sécurité) est la méthode de référence de l'ANSSI (Agence Nationale de la Sécurité des Systèmes d'Information) pour l'analyse des risques.

**Application au projet :**

**Étape 1 : Expression des besoins**
- Besoins métier : Disponibilité, intégrité, confidentialité, traçabilité
- Objectifs de sécurité : Conformité RGPD, ISO 27001, disponibilité 99.5%

**Étape 2 : Identification des scénarios de risque**
- Scénarios identifiés : Panne API Gateway, panne base de données, fuite de données, attaque
- Sources de risque : Techniques, humaines, organisationnelles, externes

**Étape 3 : Évaluation des risques**
- Probabilité d'occurrence : Faible, Moyenne, Élevée
- Impact : Faible, Moyen, Élevé, Critique
- Niveau de risque : Calculé selon la grille EBIOS

**Étape 4 : Traitement des risques**
- Acceptation : Risques faibles acceptés
- Réduction : Mesures de mitigation pour risques moyens/élevés
- Transfert : Assurance pour risques critiques
- Évitement : Abandon si risque inacceptable

**Résultat :** 11 risques identifiés avec scénarios détaillés et plan de traitement.

#### Matrice de Farmer (Impact/Probabilité)

La **matrice de Farmer** permet de visualiser les risques selon deux dimensions : l'impact et la probabilité d'occurrence.

**Application au projet :**

**Échelle d'impact :**
- **Critique :** Arrêt complet du service, perte de données, non-conformité majeure
- **Élevé :** Indisponibilité partielle, dégradation significative, impact utilisateurs important
- **Moyen :** Dégradation modérée, impact limité, récupération rapide
- **Faible :** Impact mineur, récupération immédiate, pas d'impact utilisateur

**Échelle de probabilité :**
- **Élevée :** > 50% de chance d'occurrence
- **Moyenne :** 10-50% de chance d'occurrence
- **Faible :** < 10% de chance d'occurrence

## 8. Stratégie d'élaboration du SI

### Vision du SI cible

**Objectif :** Créer une plateforme DevOps unifiée offrant :
- Une interface unique pour toutes les opérations
- Une authentification centralisée
- Une corrélation automatique des événements
- Une observabilité intégrée

### Approche de migration

**Stratégie :** Migration progressive (Big Bang évité)

**Phase 1 - MVP (6 mois) :**
- Authentification centralisée
- Gestion Kubernetes (visualisation, actions)
- Gestion Terraform (upload, visualisation, synchronisation)
- API Gateway (Kong)
- Observabilité de base (Prometheus, Grafana)

**Phase 2 - Fonctionnalités avancées (3 mois) :**
- Détection de drift Terraform
- Terminal interactif Kubernetes
- Gestion Ansible (jobs, inventaires)
- Amélioration de l'observabilité

**Phase 3 - Optimisations (3 mois) :**
- Pipelines CI/CD intégrés
- Amélioration des performances
- Optimisation des coûts
- Documentation complète

### Principes d'architecture

1. **Microservices :** Séparation des responsabilités par domaine
2. **API Gateway :** Point d'entrée unique (Kong)
3. **Event-driven :** Communication asynchrone (Kafka)
4. **Cloud-native :** Containers, Kubernetes, multi-cloud
5. **Observabilité :** Métriques, logs, traces (Prometheus, Grafana)
6. **Sécurité :** Authentification centralisée, chiffrement, audit

**S2 : Panne de la base de données PostgreSQL**
- Impact : Perte d'accès aux données utilisateurs, configurations
- RTO : 4 heures
- RPO : 1 heure (backup quotidien)

**S3 : Panne de Kafka**
- Impact : Perte de corrélation d'événements
- RTO : 2 heures
- RPO : 15 minutes (buffer)

**S4 : Panne d'un cloud provider**
- Impact : Perte d'accès aux ressources cloud
- RTO : 24 heures (bascule vers autre cloud)
- RPO : 1 heure

### Plan de Reprise d'Activité (PRA)

#### PRA - Panne API Gateway

**Objectifs :**
- RTO : 1 heure
- RPO : 0

**Procédure :**
1. Détection de la panne (monitoring Prometheus)
2. Bascule vers instance de secours (Kong en haute disponibilité)
3. Vérification de la santé des services
4. Communication aux utilisateurs

**Ressources critiques :**
- Instance Kong de secours (standby)
- Configuration sauvegardée (Git)
- Monitoring (Prometheus)

#### PRA - Panne base de données

**Objectifs :**
- RTO : 4 heures
- RPO : 1 heure

**Procédure :**
1. Détection de la panne
2. Restauration depuis backup (dernière sauvegarde quotidienne)
3. Vérification de l'intégrité des données
4. Redémarrage des services dépendants
5. Communication aux utilisateurs

**Ressources critiques :**
- Backups PostgreSQL (quotidien, rétention 30 jours)
- Instance de secours (standby)
- Procédures de restauration documentées

### Plan de Continuité d'Activité (PCA)

#### PCA - Haute disponibilité

**Objectifs :**
- Disponibilité : 99.5%
- Redondance : Multi-zone, multi-cloud

**Mesures préventives :**
1. **Infrastructure :**
   - Déploiement multi-zone (Kubernetes)
   - Load balancing (Kong, Kubernetes)
   - Auto-scaling (Kubernetes HPA)

2. **Données :**
   - Réplication PostgreSQL (master-slave)
   - Réplication Redis (sentinel)
   - Backups automatisés (quotidien)

3. **Monitoring :**
   - Alertes Prometheus (seuils définis)
   - Dashboard Grafana (vue d'ensemble)
   - Notifications (email, Slack)

#### Indicateurs de supervision

- API Gateway : 1 heure

**RPO (Recovery Point Objective) :**
- API Gateway : 0 (pas de données)

**Métriques de disponibilité :**

**Seuils d'alerte :**
- Disponibilité < 99% : Alerte critique
- Temps de réponse > 500ms : Alerte warning
- Taux d'erreur > 1% : Alerte critique
